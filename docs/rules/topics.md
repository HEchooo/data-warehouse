# 主题文档规则

## 目标

`docs/topics/*` 用来承载**业务主题级**知识，而不是通用规则，也不是统一编排说明。

## 目录结构

每个主题目录包含两个文件：

- `fact.md`
  - 主题的既定事实文档，必须存在
- `patterns.md`
  - 踩坑记录与排查经验，在真实踩坑后沉淀

## 文件职责

### `fact.md`

承载主题下报表/表级规格，默认结构为：

1. `## 报表`
2. `## 字段`
3. `## 依赖`

其中：

- `报表`：表名、粒度、Dashboard 链接（如有）
- `字段`：字段清单与口径表格
- `依赖`：上游表、中间表、脚本、DDL、编排

如果一个主题对应多张报表，可以在同一个 `fact.md` 中按报表拆成多个二级或三级小节，但每张报表仍应尽量保持 `报表 / 字段 / 依赖` 的结构。

#### `字段` 小节规则

- 如果主题需求已经完整给出，但当前只落地了一部分字段，`fact.md` 必须先列**完整需求字段清单**
- 对于已实现的部分，再单独增加 `## 已落地字段` 小节，明确：
  - 需求字段
  - 落地字段
  - 口径
- 不能因为当前只实现了部分字段，就把未实现字段从需求字段表中删掉
- 也不能把“待确认、调研中、建议方案”混写成既定事实

`fact.md` 不应承载：

- 大段排查经验（写入 `patterns.md`）
- 统一编排说明（在 `docs/references/*`）
- 跨主题规则（在 `docs/rules/*`）
- Filters 列表、更新机制、使用示例等（agent 读脚本可推断）

### `patterns.md`

用于沉淀真实出现过的问题与排查经验，适合放：

- 已发生的问题与根因
- 修复方式
- 排查顺序

如果某条经验已经跨多个主题重复出现，应从 `topics` 晋升到 `docs/rules/*`。

## 不该放在 `topics` 的内容

- 统一编排说明 → `docs/references/*`
- 跨主题复用规则 → `docs/rules/*`
- 运行环境约定 → `docs/rules/environment.md`
- 全局映射规则 → `docs/rules/mapping.md`

## 写作原则

- `fact.md` 尽量精简，只放字段口径和依赖关系
- 同一个信息只保留一个主落点，其他地方只做引用
- 多表主题可以在 `fact.md` 内用二级标题分开
- 不要因为当前只实现了部分字段，就把主题的完整需求范围写窄
